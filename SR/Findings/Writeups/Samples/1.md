# Bulletproof+ Verification Flaw Leads to Selective Miner DoS and Reward Theft

## Summary

A vulnerability in the Bulletproof+ verification logic allows an attacker to submit transactions with oversized commitment proofs to the network. These transactions are accepted into the mempool and propagated to all nodes without proper upfront validation. While these transactions are eventually rejected, the validation process consumes a disproportionate amount of CPU time and memory (in the order of a hundred milliseconds per transaction).

Malicious txs of this form pass initial mempool validation and are propagated to all nodes by design, as range proofs aren't verified at this stage. And critically, because the transactions are ultimately found to be invalid and discarded by miners/stakers, this attack can be performed at no financial cost to the attacker. Once purged from the mempool, the attacker is free to resubmit the same malicious raw txs to sustain the DoS condition at very little computational/bandwidth cost to themselves.

This can be exploited in two ways:
1.  **Network Slowdown**: An attacker can flood the mempool with these transactions to significantly slow down block processing and confirmation times across the entire network. This would eventually have a knock-on effect on difficultly adjustment, which could potentially be exploited.
2.  **Reward Theft**: More directly, a malicious miner or staker can use this exploit to gain a significant advantage over their peers, considerably increasing their share of block rewards.
## The Vulnerability

The Zano codebase implies a design limit of 32 commitments per range proof, defined by `CT::c_bpp_values_max`. This limit is correctly enforced during the proof *generation* process.

**File**: `src/crypto/range_proof_bpp.h:45`
```cpp
CHECK_AND_FAIL_WITH_ERROR_IF_FALSE(values.size() > 0 && values.size() <= CT::c_bpp_values_max && values.size() == masks.size() && values.size() == commitments_1div8.size(), 1);
```

However, this crucial check is absent in the corresponding verification functions, `bpp_verify` and `bppe_verify`. The validation logic proceeds without first confirming that the number of commitments is within the expected bounds.

**File**: `src/crypto/range_proof_bpp.h:409`
```cpp
    CHECK_AND_FAIL_WITH_ERROR_IF_FALSE(bsc.commitments.size() > 0, 2);
    CHECK_AND_FAIL_WITH_ERROR_IF_FALSE(sig.L.size() > 0 && sig.L.size() == sig.R.size(), 3);
    CHECK_AND_FAIL_WITH_ERROR_IF_FALSE(sig.r.is_reduced() && sig.s.is_reduced() && sig.delta.is_reduced(), 4);
```
The missing upper-bound check on `bsc.commitments.size()` allows the function to proceed with a proof containing far more commitments than the system was designed for.

This _could have_ led to a more severe outcome. The verification functions rely on lookup vectors whose sizes are determined at compile-time based on the assumed 32-commitment limit. Under different circumstances, processing txs with an unbounded number of commitments could have caused an out-of-bounds memory access, resulting in a segmentation fault and a daemon crash.

Fortunately, in the current code, the verification process halts gracefully due to a different check within `msm_and_check_zero_naive` before `bpp_verify` returns. The oversized commitment vector leads to the creation of an inflated `g_scalars` vector for the final multi-scalar multiplication (MSM) step. This oversized vector is caught by an explicit sanity check just before the MSM computation begins and the tx is committed to the block:

**File**: `src/crypto/msm.h:16`
```cpp
bool msm_and_check_zero_naive(const scalar_vec_t& g_scalars, const scalar_vec_t& h_scalars, const point_t& summand)
{
  CHECK_AND_ASSERT_MES(g_scalars.size() <= CT::c_bpp_mn_max, false, "g_scalars oversized");
```
While this prevents a crash (and thankfully stops oversized txs being mined), it occurs at the very end of the `bpp_verify` function, after the majority of the expensive computation and memory allocation has already been performed.

## Impact

The primary impact is the significant consumption of memory and CPU time before the transaction is invalidated. The memory footprint scales quasi-linearly with the number of commitments, which is ultimately limited only by the maximum transaction blob size (`CURRENCY_MAX_TRANSACTION_BLOB_SIZE`).

This allows a malicious proof to contain up to **889 commitments**.

The MSM size is calculated as `64 * next_power_of_2(n)`, where `n` is the number of commitments. The verification function allocates five large vectors/matrices of this size (`d`, `s_vec`, `y_inverse_powers`, `g_scalars`, and `h_scalars`), the elements of which are each 256 bits (32 bytes).

Comparing a maximum-sized valid proof with a malicious one, we have:

-   **Valid Proof (32 commitments)**:
    -   `mn` = 64 * 32 = **2048**
    -   Total memory for the five vectors: 5 * 32 * 2048 bytes = **320 KiB**

-   **Malicious Proof (889 commitments)**:
    -   `mn` = 64 * `next_power_of_2(889)` = 64 * 1024 = **65536**
    -   Total memory for the five vectors: 5 * 32 * 65536 = **10 MiB**

This represents a **32x memory amplification** for a single malicious transaction compared to the largest valid transaction.

A 'vanilla' volumetric DoS - whereby an attacker floods the mempool with large txs - is economically unfeasible, as the transaction fees outweigh the benefit of conducting the attack. However, this attack is more potent, and costs the attacker almost nothing (even when factoring in their own compute and bandwidth costs). Each malicious tx is ultimately discarded, so this work will be conducted on all miners & stakers in parallel for no benefit, and without debiting the signer's UTXO or leaving any trace on the blockchain database.
## Exploitability and Proof of Concept

Several compounding factors amplify the severity and exploitability of this issue:

1. **Oversized Proof**: As described above, txs may consume 32x the maximum memory intended, together with the associated compute-time penalty, which is estimated to be around 10x that of a typical tx (see the measurements below and PoC).
2. **Volume**: The attacker can scale this up dramatically by submitting many such txs concurrently. The scaling factor here is limited only by the attacker and network's natural bandwidth & compute bottlenecks.
3. **Propagation**: No range proof validation is performed when a transaction is first received. It is accepted into the mempool and propagated to all daemons on the network.
4. **Parallel Work**: Every miner and staker on the network will independently download and attempt to validate these costly transactions, halting all productive work while this happens.
5. **Repeatability**: Because the invalid txs are discarded, the raw tx blobs may be resubmitted by the attacker to sustain the DoS without the need to re-sign.

A proof-of-concept test is provided alongside this report, which demonstrates the attack. It is a standalone program, the code for which is based on Zano's unit test code. It measures the impact of processing these invalid txs (on compute and memory footprint) under different circumstances. 

Because the transactions pass mempool validation, no attempt has been made to conduct an end-to-end test against a running daemon: This was deemed unsafe in the absence of an isolated test network.

On my machine, the PoC showed that a single malicious transaction took **174ms** to process and invalidate, churning around **10 MiB** of memory. Submitting a few hundred (or thousand!) such transactions per minute (i.e. within the target block interval) would introduce a dramatic slowdown in block processing time network-wide.

It is noted that txs are processed in serial during PoS/PoW block generation, so catastrophic resource exhaustion (e.g. OOM) is unlikely, though this amount of heap thrashing is likely to hurt system page allocators, CPU caches and potentially page file usage, leading to further performance degradation.

While it is difficult to say with certainty if this DoS vector alone would be sufficient to halt the chain, a more targeted attack could be used to steal block rewards:

A malicious miner or staker could submit a continuous stream of these costly transactions to the network. This would impose a significant computational penalty on all other participants attempting to create a new block. The attacker, knowing which transactions are malicious, could trivially filter them out of their own mining or staking process, thereby avoiding the performance hit.

By selectively slowing down all competitors while maintaining their own efficiency, the attacker could gain a substantial advantage in winning both PoS and PoW block rewards. Quantifying this advantage would require testing on an isolated network, but it is reasonable to assume it could be considerable, potentially allowing the attacker to claim a large majority of the rewards.

## Remediation

The fix is simple - add a check to enforce the maximum number of commitments at the beginning of the `bpp_verify` and `bppe_verify` functions. This ensures that any transaction with an oversized proof is rejected immediately, before any significant computational resources are consumed.

In `src/crypto/range_proof_bpp.h` and `src/crypto/range_proof_bppe.h`:
```cpp
// Change this line:
CHECK_AND_FAIL_WITH_ERROR_IF_FALSE(bsc.commitments.size() > 0, 2);

// To this:
CHECK_AND_FAIL_WITH_ERROR_IF_FALSE(bsc.commitments.size() > 0 && bsc.commitments.size() <= CT::c_bpp_values_max, 2);
```
