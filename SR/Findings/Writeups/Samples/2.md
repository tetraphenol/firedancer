## The Bug

During transaction validation, a for-loop within `collect_rangeproofs_data_from_tx` (`blockchain_storage.cpp:6925`) iterates over the tx's commitments using a `uint8` loop variable. If a block contains a tx with more than 255 commitments, the daemon will enter an infinite loop, as the counter will wrap before it ever satisfies the exit condition.

**Location:** `src/currency_core/blockchain_storage.cpp:6925`
```cpp
for (uint8_t i = 0; i != zcrp.aggregation_proof.amount_commitments_for_rp_aggregation.size(); i++)
  agregated_proofs.back().amount_commitments.emplace_back(...);
```

The body of the loop allocates 32 bytes at a time, giving rise to a gradual memory leak which eventually takes down the daemon process with an OOM error. Until this occurs, the daemon's serialized block processing thread is stalled (measured at nearly an hour during testing). Daemons do not restart automatically. This could be abused to take all nodes offline, leading to reward monopolization, abuse of oracle outage/arbitrage, and realistic eclipse/partitioning attacks.
## Attacker Constraints

Fortunately, txpool validation prevents trivial wide-scale exploitation, though the vulnerability is still exploitable over P2P if the attacker presents a valid PoW/PoS.

The attacker is limited in that:

- Transactions submitted to the txpool are subject to output-count validation requiring that `zcrp.aggregation_proof.amount_commitments_for_rp_aggregation.size() <= CURRENCY_TX_MAX_OUTPUTS`
  - In the latest experimental branch, `CURRENCY_TX_MAX_OUTPUTS == 32`, preventing exploitation
  - Is should be noted that this isn't true of the current production build, though there may be other txpool validation measures protecting the vulnerable code
  - So it is necessary for the attacker to mine the tx into a block and share this to peers via P2P (or send it to the target daemon via RPC with `submitblock2`, which is understood not to be in scope)
- For this reason, a valid PoW or PoS must be generated
- Transaction size must be < 122,800 bytes (`CURRENCY_MAX_TRANSACTION_BLOB_SIZE`) 
- Transaction structure must be syntactically valid (vin, vout, extra, proofs)
- Aggregate proofs must be valid

However, when processing a `NOTIFY_NEW_BLOCK` relayed block from another peer (or as an explicit tx via `submitblock2`) the bug triggers before some of the validation checks would have been performed. This allows the attacker to simplify their attack, using:

- Fake range proofs
- Fake key images
- Fake ring members
## Exploitation

Two forms of PoC are provided with this report:

1. A unit test: **uint8_overflow_poc** (see uint8_overflow_test.cpp)
  - Generates a transaction containing 256 commitments (or at least, something that looks enough like a transaction to satisfy all the block validation logic)
  - Serializes the tx blob to disk for inspection (or for use by the other PoC)
  - Creates an empty block and adds the malicious tx
  - Solves PoW
  - Adds the block to events for processing
  - Test process the processes the block, hangs for a while and then crashes with a bad alloc
2. A standalone PoC (**gen_oversized_tx** and **submit_p2p**)
  - `gen_oversized_tx` generates a malicious tx as above and writes it to disk
  - `submit_p2p` then:
    - Loads and parses the tx
    - Queries a local daemon for a block template
    - Mines a PoW block with the malicious tx as an explicit transaction
    - Peers with the target daemon (`COMMAND_HANDSHAKE` message)
    - Submits the block to the target daemon using `NOTIFY_NEW_BLOCK`
  - The target daemon process will hang and crash as before
  - Note that validation (and DoS) occurs _before_ the block is committed to local storage and so it is not relayed on to other peers. So this attack can only be used to DoS immediate peers

An attacker seeking to DoS the entire network would need to work a little harder. Something along the lines of:
- Write some code to map out the peer network (using `COMMAND_HANDSHAKE_T` to learn peers' neighbors iteratively)
- Establish a peer relationship with as many daemons as possible
  - Many will refuse the initial peering request as they are saturated
- Mine a malicious block and deliver it concurrently to all peers (`NOTIFY_NEW_BLOCK`)
- Poll the neighbors of these DoSed daemons with further peering requests to occupy the empty slot once the connection times out
- Repeat the process. Any daemons that come back online can be relatively easily taken down again with sufficiently aggressive peer requests as they won't have an opportunity to full up on peers 
  - The malicious daemon could feed a malicious block into sync responses to recovering peers (`NOTIFY_RESPONSE_GET_OBJECTS`) where possible to eliminate the race condition

Network-wide exploitation would likely require a few PoWs to be generated to be successful, but the slow memory leak works in the attacker's favor, and winning this race would get easier as competition progressively drops out:

- During testing, the daemon's block processing was seen to stall for a long time prior to the daemon crashing - upward of 45 minutes on my test machine
- Systems with more memory will take longer to fail
- Most daemons probably won't restart until the operator discovers they have stopped and manually intervenes
- Even if any daemons are configured to auto-restart, they would still have been down for a large majority of time while the memory leak gradually exhausts RAM in between restarts (because block processing ceases as soon as the malicious block is validated)

A well-funded attacker could seek to use stake to gain an advantage and use PoS mining to produce valid blocks, rather than PoW.

## Impact

Ultimately, a sustained DoS effort from a suitably equipped attacker would halt the network until a software fix was developed and manually deployed to a significant number of daemon operators. During this time, the attacker would be free to:

* Monopolize the network outage to claim mining rewards
* Capitalize on arbitrage/oracle-outage opportunities for financial gain on exchanges relying on chain uptime
* Manipulate network topology, conducting eclipse or network partitioning attacks against recovering peers

## Proof Of Concept 1 - Unit Test

All PoC files are provided in the accompanying gist: https://gist.github.com/monster-punch/a4163a115f36593d3743c5805f2ea7b0

The first PoC - `uint8_overflow_poc` is built as a unit test into the existing `coretests` suite: place it in the `tests/core_tests` directory and be sure to reference it in `chaingen` before building with `BUILD_TESTS=TRUE`.

In order to build the test, it is necessary to temporarily patch the following definition in `range_proofs.h`:

```
- typedef bpp_crypto_trait_zano<bpp_ct_generators_UGX, 64,  32> bpp_crypto_trait_ZC_out;
+ typedef bpp_crypto_trait_zano<bpp_ct_generators_UGX, 64,  256> bpp_crypto_trait_ZC_out;
```

Remember to revert the patch after building the test suite (or build the daemon beforehand)!

The test can then be run as follows:

```
user@ubuntu:~/Zano/zano/build$ ./tests/coretests --run-single-test uint8_overflow_poc
```

Because the memory leak is slow (just 32 bytes per iteration), it takes some time for the eventual OOM/failed alloc and unhandled exception to trigger - approaching an hour after exploitation on my test machine. During this time, block processing is stalled on the daemon, but other threads continue to operate (peer exchange, RPC etc.). The fault condition can be observed by monitoring the daemon's resource usage following delivery: CPU will remain at around 100% while memory gradually creeps upward.
